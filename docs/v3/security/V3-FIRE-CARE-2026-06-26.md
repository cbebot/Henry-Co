## Findings

### XD-F03 â€” (cross-division) the retired-domain god-mode seeds are still live (CRITICAL)
- **Category:** Privilege escalation â†’ money-control account takeover. This is finding **F-03** from `V3-FIRE-MARKETPLACE-2026-06-26.md` (PR #337); this pass **re-verified it on prod** because the company-name correction made it acute.
- **Pre-fix reproduction (live):**
  ```sql
  select count(*) from marketplace_role_memberships
   where normalized_email ilike '%henrycogroup.com' and is_active;
  -- â†’ 7   (henrycogroup_active = 7, henrycogroup_total = 7)
  ```
  The 7 rows confer owner/finance/admin/moderation/operations/support/vendor, all keyed to a `henrycogroup.com` email; `getMarketplaceViewer` (`apps/marketplace/lib/marketplace/auth.ts:90-103`) binds roles by `normalized_email` match. Owner confirmed `henrycogroup.com` is **permanently retired** â‡’ re-registerable â‡’ an attacker can claim the domain + mailbox and inherit money-control. **A prior note asserted "zero henrycogroup.com rows on prod" â€” the live count refutes that; the seeds were never removed.**
- **Fix (owner-gated prod write):**
  ```sql
  update public.marketplace_role_memberships
  set is_active = false
  where normalized_email ilike '%henrycogroup.com' and is_active;   -- expect 7
  ```
  plus the `auth.ts` email-binding fix (don't confer privileged roles by email match) â€” see PR #337 F-03 and `apps/marketplace/lib/marketplace/authorization.ts` from PR #338 (note: PR #338 left the email-OR binding untouched as a "latent residual" on the premise of zero rows, which this probe disproves).
- **Post-fix reproduction:** re-run the count â†’ `0`.
- **Regression test:** CI invariant â€” no active `*_role_memberships` row carries a `henrycogroup.com` email; `getMarketplaceViewer` unit test (email-only row â‰  role grant).

### CARE-1 â€” `care_bookings` "view by email" RLS policy (MEDIUM, latent)
- **Category:** IDOR (the #333 class) at the RLS layer. CVSS-ish: latent, email-verification-gated, 0 live instances.
- **Pre-fix reproduction (live):** policy dump shows three `care_bookings` SELECT policies â€” `auth.uid()=customer_id` (correct), the service_role policy, and:
  ```
  "Users can view bookings by email"  SELECT  using ( lower(email) = lower(auth.jwt() ->> 'email') )
  ```
  An authenticated user can read any booking whose `email` equals their JWT email, regardless of `customer_id`. This is the exact pattern behind the #333 incident (ownership resolved by email, not a stable FK). Live exposure today:
  ```sql
  select count(*) from care_bookings b join auth.users u
    on lower(u.email)=lower(b.email)
   where b.email is not null and b.customer_id is distinct from u.id;
  -- â†’ 0   (email_collision_count = 0; 51/54 bookings carry customer_id, 3 guest/unlinked)
  ```
  So it is **not currently exploitable** (no booking's email matches a different account; signup requires email verification, so a user can only match their own verified email). But it is a latent IDOR: a guest booking made with an address that a different person later registers, or any email reuse, becomes cross-readable.
- **Proposed fix (held migration `01`):** drop the email policy; rely on `customer_id` (+ the service_role `/track` path for guests). The app's `care-recent-bookings.ts:46` (`.ilike("email", userEmail)` via the admin client) is the same class on the app side â€” switch to `.eq("customer_id", user.id)`.
- **Post-fix:** re-run the collision probe (stays 0) and confirm an authed user reading a foreign booking by shared email returns 0.

### CARE-2 â€” `/api/care/track` optional-phone capability URL (MEDIUM)
- **Category:** Smart-leak (capability URL). Not enumerable â‡’ MEDIUM, not CRITICAL.
- **Pre-fix reproduction (source):** `apps/care/app/api/care/track/route.ts` (service_role, `force-dynamic`) resolves a booking by `tracking_code` (`.eq("tracking_code", code)`) and returns `customer_name, phone, pickup_address, special_instructions, quoted_total, balance_due, payment_status` plus, when present, POD `gps_lat/gps_lng/recipient_name`. The phone check is `if (phone) { ... }` (route L74) â€” it only runs when a phone is supplied, so the tracking code is a sole bearer token. Entropy census (live): codes are `TRK`-prefixed, **12â€“14 chars** (`min_len=12, max_len=14`, n=54) â€” **not** the `HC-YYYYMMDD-{6}` the static map guessed, and **not** mass-enumerable like marketplace's 900/day. So exposure is via a *leaked/shared* code (referrer, logs, forwarded link), not enumeration.
- **Proposed fix:** require a successful phone match (or an authenticated owner) before returning PII/GPS; don't treat a missing phone as a pass. (App-layer; no migration.)
- **Post-fix reproduction:** GET `/api/care/track?code=<valid>` with no phone â†’ returns status only (no name/phone/address/GPS) unless phone matches or the viewer owns the booking.

### CARE-3 â€” fractional naira reached the money tables (MEDIUM, LIVE)
- **Category:** Money integrity. Operator-gated (manual review), so not self-serve inflation â€” but proven to manifest.
- **Pre-fix reproduction (live):**
  ```sql
  select count(*) from care_finance_ledger where amount <> round(amount);  -- â†’ 3
  select count(*) from care_payments       where amount <> round(amount);  -- â†’ 3
  ```
  CARE money is whole-naira; `apps/care/lib/payments/verification.ts:sanitizeAmount` is `Math.max(0, Number(v))` with **no rounding**, so a client-claimed fractional `amount_paid` (or a null `amountApproved` defaulting to it) is preserved into the ledger/payments. 3 live rows in each confirm the path is reachable.
- **Proposed fix:** round at the boundary (`Math.round`) in `sanitizeAmount` and reject/normalize non-integer naira; backfill the 6 rows under finance review. (App-layer + data cleanup; no schema migration.)
- **Post-fix:** both counts â†’ 0.

### CARE-4 â€” no FORCE RLS on care money/PII tables (MEDIUM, hardening)
- **Pre-fix reproduction (live):** posture sweep â€” `care_bookings`, `care_payments`, `care_payment_requests`, `care_finance_ledger`, `care_expenses`, `care_notification_queue`, `care_order_items` all `relforcerowsecurity=false` (only `care_site_settings` + `care_pricing_items` are forced). All writes flow through `service_role` (bypasses RLS), so a leaked key or a forgotten owner-filter has no DB backstop.
- **Proposed fix (held migration `02`):** `ALTER TABLE â€¦ FORCE ROW LEVEL SECURITY` on the care money/PII tables. Verify staff/service read paths (which use service_role) are unaffected first.

### CARE-5 â€” anon/PUBLIC-executable money + code functions (LOW, hardening)
- **Pre-fix reproduction (live):** `care_recalculate_booking_totals(p_booking_id)` and `make_care_tracking_code()` have `acl` containing the empty-grantee `=X/` (PUBLIC) entry and `has_function_privilege('anon',â€¦,'EXECUTE')=true`. `care_recalculate_booking_totals` is `SECURITY INVOKER`, so anon's underlying write to `care_bookings` is still blocked by RLS (no anon write policy) â€” but a money-recompute and the tracking-code generator should not be PUBLIC-executable. (`create_care_booking`/`track_care_booking` are anon-executable **by design** for guest booking/tracking; `is_staff_in` is a safe boolean self-check.)
- **Proposed fix (held migration `03`):** `REVOKE EXECUTE â€¦ FROM public, anon, authenticated` on `care_recalculate_booking_totals` and `make_care_tracking_code` (the PUBLIC grant is additive â€” revoke from `public` too, not just `anon`).

### CARE-6 â€” `care_settings` exposes company payment-receiving bank fields (LOW, by-design)
- **Category:** Informational â€” **public-by-design**, per calibration **not** a vulnerability.
- **Live facts:** `care_settings` has a `USING(true)` SELECT policy for `{anon,authenticated}` and is anon-readable (`anon_care_settings_rows=1`); it carries `company_account_name/number/bank_name` + `payment_account_name/number/bank_name`. These are the **company's own payment-receiving accounts** that a bank-transfer checkout must display to customers â€” i.e. published payment instructions, not customer/vendor/credential data. (The genuinely sensitive `care_payment_requests` bank fields are in a separate `RLS-on+0-policy` lockbox proven to deny anon.)
- **Optional hardening:** if the company prefers its raw receiving account not be blanket-scrapeable, serve it from a column-restricted view or behind an active-booking/checkout context rather than a public settings read. No fix shipped (by-design).
