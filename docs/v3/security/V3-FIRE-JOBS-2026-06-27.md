- **Entity:** Henry Onyx Limited (codename `HenryCo` / `@henryco/*`; only valid domain **henryonyx.com**).
- **Division:** jobs (`apps/jobs` — recruiting: applications, hiring pipelines, candidate↔employer DM (WS‑5), offers, interviews).
- **Live DB audited:** Supabase project `rzkbgwuznmdxnnhmjazy` ("HENRY ONYX").
- **Date:** 2026-06-27
- **Method:** static map (3 parallel subagents: RLS/SECDEF catalog; candidate-PII/offers/pipeline authz; WS‑5 messaging + masking + money) → live read-only probing as the **real `anon` and `authenticated` (stranger) roles** (`SET LOCAL ROLE` + `request.jwt.claims`), never `service_role`. ## Executive summary

**Jobs is the highest-severity division of the sweep — but the severity is entirely in the application-layer authorization, not the data layer.** Two key live facts shape every grade:

1. **The data layer holds, and the scariest hypothesis was refuted by probe.** `jobs_messages` and `jobs_conversations` are on the `supabase_realtime` publication (with `replica identity full`), which raised a candidate-CRITICAL "any authenticated user streams every hiring DM." **Refuted:** both tables — and every base table (`jobs_applications`, `jobs_interviews`, `jobs_hiring_pipelines`, `jobs_contact_masks`, `jobs_moderation_queue`) — are **RLS-on + 0 policies** (deny-all). Live probes: anon read = **0**, authenticated-stranger read = **0** for all of them. Realtime re-applies the (nonexistent) SELECT policy → a non-participant subscription receives nothing. Candidate resume URLs / emails are not anon/cross-user readable at the DB layer.
2. **There is no production data yet** (`applications=0, conversations=0, messages=0, pipelines=0, employers=0`). So the app-layer authz defects below are **confirmed code defects that are not *currently* exploitable** — they are **pre-launch / pre-data launch-blockers**: each becomes live the instant the division has real applications/conversations.

Every base-table access in the app runs through `createAdminSupabase()` (**service_role, bypasses RLS**), so for these routes **RLS is not a backstop — the app-layer check is the only control.** That is where the defects cluster.

## Findings

### JOB-1 — `/api/hiring/interviews` unauthenticated interview injection (CRITICAL, pre-data)
- **Category:** Missing authentication on a mutating endpoint → in-platform phishing. CVSS `AV:N/AC:L/PR:N/UI:R/S:C/C:L/I:H/A:L` ≈ **8.x**.
- **Asset:** `apps/jobs/app/api/hiring/interviews/route.ts` (whole handler) → `scheduleInterview` (`apps/jobs/lib/jobs/hiring.ts:564`). `jobs_interviews` **exists on prod**.
- **Pre-fix reproduction (source — verified first-hand):** the POST handler parses JSON and calls `scheduleInterview({ applicationId, title, scheduledAt, meetingUrl, notes, ... })` with **no `getJobsViewer`, no session, no membership/ownership check** (grep for `getJobsViewer|auth` in the file → none). `proxy.ts` is **passive auth** ("verifies/refreshes the Supabase session and tags the request") — it does **not** block unauthenticated requests, so there is no middleware backstop. `scheduleInterview` inserts directly via the service_role client.
  - **Impact:** an unauthenticated attacker creates a `jobs_interviews` row on any `applicationId` with an attacker-controlled `meetingUrl`, which renders on the candidate's dashboard as a genuine "Henry Onyx interview" link → credential-harvest / phishing, plus unauthenticated write-amplification. *Pre-data:* with 0 applications today there is no valid `applicationId` to target, but the missing auth is unconditional and becomes exploitable the moment applications exist (and ids leak via employer URLs/emails).
  - Repro: `curl -X POST https://jobs.<host>/api/hiring/interviews -H 'content-type: application/json' --data '{"applicationId":"<id>","title":"Final interview","scheduledAt":"2026-07-01T10:00:00Z","meetingUrl":"https://evil.example/phish","notes":"join with your Henry Onyx password"}'` with no cookie → expect 200 + row created.
- **Proposed fix:** add `getJobsViewer()` and the shared employer-ownership check (resolve `applicationId → pipeline → employer_id`, require it ∈ `viewer.employerMemberships`, or staff); reject unauthenticated requests. (App-layer.)
- **Post-fix reproduction:** unauthenticated POST → 401; an employer not owning the application → 403.
- **Regression test:** route test — unauthenticated + cross-employer POST both denied.

### JOB-2 — Cross-conversation message injection / employer spoofing (HIGH, pre-data)
- **Asset:** `apps/jobs/app/api/hiring/messages/route.ts:110-123` (`resolveParticipantRole`) and `hiring/messages/flag/route.ts:108-114` (`resolveFlaggerRole`). `jobs_conversations`/`jobs_messages` exist on prod.
- **Pre-fix reproduction (source):** the employer branch returns `{role:"employer"}` when `pipelineRow` exists **and** `viewer.employerMemberships.length > 0` — **it never selects or checks `jobs_conversations.employer_id`**. (The candidate branch *is* strict: `conversationCandidateId === viewerId`; and both *read* pages enforce `employer_id === viewerId` — only the write/flag paths are loose. The route comment admits it's "conservative until production conversations exist.") So any user with ≥1 employer membership (trivially self-granted via `createEmployerProfile`) can POST into / flag **any** company's candidate thread; the message renders to the candidate as `sender_type="employer"` under the real employer name → impersonation/phishing. (Body masking still applies, so this is integrity/spoofing, not a contact leak.)
- **Proposed fix:** in `resolveParticipantRole`/`resolveFlaggerRole`, select `employer_id` and require `employer_id === viewerId` (and the membership slug matches the pipeline's `employer_id`/`company_id`); drop the bare `employerMemberships.length > 0` gate.
- **Probe:** as Employer A, `POST /api/hiring/messages {conversationId:<B's convo>, body:"…"}` → a 200 proves the gap.

### JOB-3 — `jobs/pipeline/move` cross-tenant stage manipulation (HIGH, pre-data)
- **Asset:** `apps/jobs/app/api/jobs/pipeline/move/route.ts:72-96` → `updateApplicationStage` (`hiring.ts:602`). `jobs_applications` exists on prod.
- **Pre-fix reproduction (source):** non-staff path checks `employerMemberships.length>0` and that the application row *exists* — it **never** ties `appRow.pipeline_id → jobs_hiring_pipelines.employer_id` to the viewer. `updateApplicationStage` does an unscoped `update … .eq("id", applicationId)`, and stage strings are forwarded as-is (bypassing the per-pipeline stage allowlist). Any employer-member can move any other employer's applicant to any stage (e.g. mass-reject a competitor's funnel).
- **Proposed fix:** resolve pipeline→employer and require membership match before `updateApplicationStage`.

### JOB-4 — `jobs/offers` cross-tenant offer creation (MEDIUM, dormant)
- **Asset:** `apps/jobs/app/api/jobs/offers/route.ts:37-92` → `issueOfferLetter`. **`jobs_offer_letters` is NOT on prod** (committed-not-applied) — the route would fail today, so this is **dormant**.
- **Defect (will be HIGH once applied):** auth checks "staff OR any employer membership", then loads the application by client-supplied id and **never** matches `pipeline.employer_id` to the viewer's membership before writing an offer (salary terms + candidate-name event) onto another tenant's application.
- **Fix:** add the membership↔pipeline ownership check **before** applying the offer-letter migration; validate `baseSalaryMinor` (see JOB-9).

### JOB-5 — Interview-room notes IDOR (MEDIUM, dormant)
- **Asset:** `apps/jobs/app/api/jobs/interviews/rooms/[roomId]/notes/route.ts:72-101` → `updateInterviewRoomNotes`. **`jobs_interview_rooms` is NOT on prod** (committed-not-applied) → **dormant**. The code comment itself admits "we don't yet attempt strict employer-slug-pipeline join."
- **Defect (HIGH once applied):** any employer-member can overwrite up to 8,000 chars of private interview notes on any room by `roomId` (existence check only). The join data is already selected — it just isn't compared. **Fix:** compare resolved `pipeline.employer_id` to the viewer's memberships before applying the migration.

### JOB-6 — Candidate documents at permanent public URLs (MEDIUM)
- **Asset:** `apps/jobs/app/api/candidate/documents/route.ts` → `uploadCandidateAsset` (`lib/jobs/write.ts:480`) → `lib/cloudinary.ts:51-128`. Upload is correctly owner-scoped (no read-IDOR), but assets are uploaded as Cloudinary `image/upload`/`raw/upload` (default delivery) and the public `secure_url` is persisted to `customer_documents.file_url` — **no `access_mode:authenticated`/`type:private`, no signed URLs**. Resumes and ID documents live at permanent, world-readable URLs (security-by-obscurity; the only protection is a random 12-hex `public_id`). Any URL leak (referrer, logs, forwarded link) = permanent unauthenticated PII access with no revocation.
- **Fix:** upload candidate docs as `authenticated`/`private` and serve via short-lived signed URLs through an ownership-checked route.

### JOB-7 — Cover-note / subject masking bypass (MEDIUM)
- **Asset:** cover note stored unscreened (`lib/jobs/write.ts:713,745-757`), mapped raw (`hiring.ts:75`), rendered to the employer **unmasked** (`app/employer/applicants/[id]/page.tsx:106`, `app/employer/hiring/[pipelineId]/[applicationId]/page.tsx:156`). Same for `jobs_conversations.subject` (candidate page `:110/:116`).
- **Defect:** the WS‑5 contact-masking screen is **verified closed on the `jobs_messages` DM** (block/mask before persist, both directions, sole write path, read-path re-mask — PASS). But the cover note is a *different* contact channel that **skips the screen entirely** — a candidate can put a phone/email/WhatsApp in the cover note and reach the employer before any DM, defeating the masking system's purpose (anti-disintermediation + unmediated PII exchange).
- **Fix:** run `screenMessageBody` (block/mask) on cover notes at submit, and `maskContactsForDisplay` on cover note + conversation subject at render.

### JOB-8 — Hardening: FORCE-RLS + the policy-less realtime tables (LOW)
- No jobs table is `relforcerowsecurity=true`. Recommend FORCE on `jobs_applications`, `jobs_interviews`, `jobs_conversations`, `jobs_messages`, `jobs_contact_masks`, `jobs_moderation_queue`, `jobs_hiring_pipelines` (held migration `01`).
- `jobs_messages`/`jobs_conversations` are realtime-published but have **0 policies** — secure now (deny-all), but the live-message feature has **no backing participant policy**, so realtime delivers nothing to legitimate participants either. **Do not "fix" the UX by adding a permissive policy** — add a participant-scoped SELECT policy (held migration `02`) so realtime is both functional *and* leak-proof. This is the forward-looking guard for the candidate-CRITICAL that the probe refuted today.

### JOB-9 — Lower-tier (LOW)
- `jobs_employer_subscriptions` enforcement is **fail-open** (`no-table`/`no-record`/`query-error`/unknown → `allowed:true`, `lib/jobs/employer-subscription.ts:43,49,70`) — a DB hiccup lets unsubscribed employers post. Fix: fail closed.
- Offer `baseSalaryMinor` has no integer/non-negative/range validation (`offers/route.ts:123` → stored as-is). Fix: `Number.isInteger && >=0 && <=CAP`.
- `getEmployerMembershipsByUser` (`lib/jobs/data.ts:1133`) unions memberships by `user_id` **and** by `metadata.normalizedEmail` match — the recurring email-OR class; broadens the blast radius of JOB-2/3/4. Prefer `user_id` binding.
