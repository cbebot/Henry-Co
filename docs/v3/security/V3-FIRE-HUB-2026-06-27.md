## Findings

### HUB-1 ÔÇö MEDIUM ÔÇö `is_owner()` is SECURITY INVOKER, causing infinite RLS recursion on `owner_profiles`
**Asset:** `public.is_owner()` + `owner_profiles` RLS.
**Evidence (prod, verbatim):** `is_owner()` is `LANGUAGE sql STABLE`, `SET search_path TO 'public','pg_catalog'`, `prosecdef = false`. Body: `select exists(select 1 from public.owner_profiles op where op.user_id = auth.uid() and op.is_active and op.role in ('owner','admin'))`. The `owner_profiles` policy `owner_profiles_select_self_or_owner` is `auth.uid() = user_id OR is_owner()`.
**Repro (live):** an authenticated-stranger `SELECT ÔÇĞ FROM owner_profiles` ÔåÆ `ERROR: stack depth limit exceeded`. For any non-owner row, the `auth.uid()=user_id` disjunct is false, so `is_owner()` is evaluated; its inner read of `owner_profiles` re-applies the same SELECT policy ÔåÆ re-invokes `is_owner()` ÔåÆ unbounded recursion.
**Why it's not worse today:** the app reads `owner_profiles` via the service-role client (RLS bypassed), so the recursion is invisible to normal flows. But authenticated Data-API reads of `owner_profiles` 500 instead of returning empty, and any future RLS path that consults `owner_profiles` for authenticated users inherits the DoS.
**Fix:** `SECURITY DEFINER` (held migration `01`). CVSS-ish: low confidentiality/integrity impact, availability/correctness defect ÔåÆ MEDIUM.

### HUB-2 ÔÇö MEDIUM (latent) ÔÇö `owner_profiles_update_own` does not pin privileged columns
**Asset:** policy `owner_profiles_update_own` (UPDATE).
**Evidence (prod):** `USING (auth.uid()=user_id)` and `WITH CHECK (auth.uid()=user_id)` ÔÇö no constraint on `role` or `is_active`. An owner_profiles row holder can UPDATE their own row to change `role` ('admin'ÔåÆ'owner') or `is_active`.
**Live exposure:** none today ÔÇö exactly one owner_profiles row exists, already `role='owner'`, so there is no privilege to gain. The hole opens when a second lower-tier ('admin') row is added: that admin could self-promote to 'owner'.
**Fix:** column-revoke `UPDATE (role, is_active)` from `authenticated` (held migration `02`); privileged changes stay on the `owner_profiles_owner_write`/is_owner() path. ÔåÆ MEDIUM (latent).

### HUB-3 ÔÇö MEDIUM ÔÇö `hq_ic_members` self-management lets staff gain WRITE into owner broadcast threads
**Asset:** `hq_internal_comm_thread_members` policies `hq_ic_members_insert` / `hq_ic_members_update`.
**Evidence (prod):** insert `WITH CHECK ((user_id = auth.uid()) AND hq_ic_can_read_thread(thread_id))`; update `USING/CHECK (user_id = auth.uid())`. Neither constrains `role`. An `all_owners` thread is readable by any active staff (`hq_ic_can_read_thread` ÔåÆ owner OR active staff membership), so a staffer can self-INSERT a membership row with `role='member'`/`'owner'`, or self-promote an `observer` ÔåÆ `member`. `hq_ic_can_write_thread()` then returns true (role distinct from `observer`) ÔåÆ the staffer can **post messages into owner broadcast/announcement threads** they could previously only read.
**Method note:** confirmed from policy text; **no write probe** was performed (READ-ONLY mandate). HQ-comms data is effectively pre-launch.
**Fix (held migration `03`, secure default):** self-join allowed only as read-only `observer`; column-revoke `UPDATE (role)` from `authenticated`. ÔÜá Architect must confirm whether active staff are *meant* to reply in `all_owners` threads ÔÇö if so, grant writer membership via service-role, not self-join. ÔåÆ MEDIUM.

### HUB-4 ÔÇö MEDIUM (latent) ÔÇö `requireOwner` email-fallback (email-OR class)
**Asset:** `requireOwner`/owner-gate (TS), which (per the owner-surface map) falls back to matching `owner_profiles` by email when no `user_id` row matches.
**Evidence (prod):** the single `owner_profiles` row is `user_id`-bound and there are **0 `user_id=NULL` owner rows**, so the email-fallback is never the resolving path today.
**Why it still matters:** this is the same systemic email-OR binding flagged in marketplace (henrycogroup, since neutralized), studio, jobs, learn, and account ÔÇö a `user_id=NULL` owner row with a claimable email would let whoever registers that email become owner. Remove the email-fallback; require `user_id` for the owner binding. ÔåÆ MEDIUM (latent). **Operational note:** the sole owner row is on a personal `@gmail.com` address ÔÇö recommend enforced 2FA on that account (it is the keys to the kingdom), and migrating owner identity to a henryonyx.com address.

### HUB-5 ÔÇö LOW ÔÇö Owner-reporting cron fail-open
The owner-reporting cron authorizes when `CRON_SECRET` is unset **and** `NODE_ENV !== 'production'`. Inert on Vercel (secret present + prod), but the fail-open branch should be removed so a missing secret fails **closed**. Same class as the care/learn/property cron notes.

### HUB-6 ÔÇö LOW ÔÇö Over-broad `all_owners` read + non-constant-time cron compare
- `all_owners` (division NULL) threads are readable by **any active staff**, not just owners ÔÇö by design per code comments, but verify the intended audience matches the "owners" label before owner-only content is posted.
- The cron secret comparison is not constant-time (timing side-channel; low practical risk over HTTPS). Use a constant-time compare.

### HUB-7 ÔÇö INFO ÔÇö Governance gaps
- **`owner_profiles`, `company_divisions/homepages/homepage_faqs/pages/people/site_settings` have no `CREATE TABLE` in source control** (applied-not-in-repo, consistent with V3-RECONCILE-01). Owner-tier identity tables with no DDL in git is a governance gap ÔÇö their RLS/policies can only be verified live (done here). Recommend capturing them into the repo.
- The **owner inbox** (`received_emails`/`received_email_attachments`/`owner_inbox_is_owner()`) is absent from this branch but **live on prod** (V3-OWNER-INBOX-01, built off a separate worktree). Probed here: owner-gated, authenticated-stranger read = **0** (92 stored emails). Sound; flagged only so it isn't assumed "covered by repo text."
